\subsection{Xamarin.Forms: 3 Native UIs, 1 Shared Code Basis}
Bei Xamarin.Forms handelt es sich um ein .NET-basiertes UI Toolkit, mit dem Oberflächen entweder in
C\# oder einem speziellen XAML-Dialekt generisch definiert werden. Während des Kompiliervorgangs
entstehen aus dem gemeinsamen Code plattformspezifische, native Benutzeroberflächen. Jeder Seite und
deren Steuerelementen werden plattformspezifische native UI Elemente zur Laufzeit zugeordnet. Z.B.
ein Xamarin.Forms Entry wird zu einem UITextView auf iOS, EditText auf Android und TextBox auf
Windows Phone. Xamarin.Forms unterstützt die Plattformen iOS, Android und Windows Phone, aber (noch)
nicht Windows Store Apps. Mit Xamarin.Forms können Entwickler deutlich weniger Code schreiben und dadurch wird der jeweilige Mehraufwand pro zu unterstützende Plattform geringer. Allerdings gibt es (noch) keinen UI Designer bei Xamarin.Forms
und man muss die Benutzeroberflächen aus dem Code gestalten.
%%\includegraphics[scale = 0.5]{graphics/XamarinForms1.png}
\\Es gibt zwei
unterschiedliche Lösungsansätze für die plattformübergreifenden Projekte:
\begin{itemize}
      \item Shared Projects
      \item Portable Class Libraries (PCL)
   \end{itemize}
Bei dem Shared Projects Ansatz wird der enthaltene Code für die jeweilige Plattform kopiert und
erneut übersetzt. Auf diese Weise wird der Code direkt in die Assembly des referenzierenden Projekts
integriert und es wird keine neue Assembly erzeugt. So lässt sich der Code dieser Assembly auf
mehreren Plattformen wiederverwenden und zugleich über Präprozessordirektiven oder partielle
Methoden und Klassen in den nativen Projekten auf plattformspezifische Anforderungen eingehen, ohne
zusätzliche Abstraktionsschichten für die nativen Funktionen einbauen zu müssen. Allerdings muss
hier erwähnt werden, dass bei diesem Lösungsansatz die Übersicht im Code schnell verloren geht und
dadurch die Wartbarkeit des Codes erschwert wird. 
\\Bei dem zweiten Lösungsansatz (PCL) bietet sich hingegen die Möglichkeit, mit gemeinsamer
Funktionalität ausgewählter Zielplattformen gemeinsamen Code in einer Assembly abzubilden. Dadurch
wird ein möglich effizientes Code Sharing erreicht.

\subsubsection{View und Layouts}
Xamarin.Forms stellt eine API bereit, die für das Erstellen von Benutzeroberflächen mit
Steuerelemente und Layouts genutzt wird. Die Xamarin.Forms Steuerelemente werden zur
Laufzeit den entsprechenden nativen Steuerelementen zugeordnet. Folgende vier Klassen werden für
die Gestaltung der Benutzeroberflächen einer Xamarin.Forms App verwendet:
\begin{itemize}
  \item \textbf{View} - repräsentiert Steuerelemente, wie:
  \begin{itemize}
    \item \emph{Label} - analog zu der herkömmlichen Software Entwicklung ein "`read only"' Display
    Steuerelement.
    \item \emph{Entry} - einzeiliges Eingabetextfeld.
    \item \emph{Button}.
    \item \emph{Image}.
    \item \emph{ListView} - eine scrollbare Liste von Elemente (Cells).
  \end{itemize}
  \item \textbf{Page} - repräsentiert eine einzelne Seite auf dem Bildschirm, analog zu einer
  Activity bei Android, iOS ViewController oder Windows Phone Page. Xamarin.Forms bietet fünf
  verschiedene Arten von Pages an:
    \begin{itemize}
        \item \emph{Content Page} 
        \item \emph{MasterDetailPage} 
        \item \emph{NavigationPage} 
        \item \emph{TabbedPage} 
        \item \emph{CarouselPage} 
    \end{itemize}
  \item \textbf{Layout} - Spezielles Untertyp von View. Wird als Container für andere Layouts oder
  Views verwendet. Es gibt zwei Arten von Layouts in Xamarin.Forms:
  \begin{itemize}
    \item \emph{Managed Layouts} - ein Beispiel ist das StackLayout von Xamarin.Forms. Managed
    Layouts sind für die Positionierung und die Größenanpassung ihrer Kinder zuständig (ähnlich wie das
    CSS-Box-Model bei der Web Entwicklung).
    Applikationen sollten nicht direkt die Größe und Position der Kind-Steuerelemente eines StackLayouts anpassen
    können.
    \item \emph{Unmanaged Layouts} - im Gegensatz zu managed Layouts kümmern sich Layouts, wie z.B.
    AbsoluteLayout, nicht um die Positionierung ihrer Kinder auf dem Bildschirm.
  \end{itemize}
  \item \textbf{Cell} - ein Element einer Liste oder Tabelle. Beschreibt wie jedes Element einer
  Liste abgebildet werden soll.
\end{itemize}
Views werden immer in Layouts untergebracht.\newline[\cite{XAM}]
\section{NuGet und Xamarin Component Store}\label{nugets}
Xamarin bietet die Möglichkeit Komponente zu jeder App direkt aus der IDE hinzuzufügen.
Es gibt zwei Arten von Komponenten, die zu einem
Xamarin-Projekt angebunden werden können, nämlich Xamarin Components und NuGet Packages.
Dementsprechend gibt es Xamarin Component Store und NuGet Gallery, in denen eine große Vielfalt
von "`open source"' Bibliotheken zur Verfügung steht. \\Somit lassen sich diverse Controls, Web
Service APIs und vieles mehr benutzen, ohne die Features selber entwickeln zu müssen.
Es können auch populäre Backends wie Microsoft Azure, Parse, Salesforce, SAP usw. in die APP integriert werden,
sowie auch Security Features wie Authentifizierung und Verschlüsselung.

\section{Fazit}
Xamarin.iOS und Xamarin.Android verschaffen direkten Zugriff auf die plattformspezifischen APIs und
dadurch lassen sich Apps erstellen, die spezielle Interaktionen erfordern. Solche Apps haben
die Funktionalität und die Leistung einer nativen Anwendung.\\Hingegen ist der Ansatz von
Xamarin.Forms besonders gut für Apps geeignet, die wenig plattformspezifische Funktionalität
aufweisen oder solche, bei denen keine spezielleren Benutzeroberflächen erfordert
werden.
Die von Xamarin.Forms angebotenen Steuerelemente sind meistens
ausreichend, so dass ein benutzerdefinierter Renderer selten benötigt wird. Ein Beispiel
wäre die Hintergrundfarbe einer markierten Zeile eines ListViews. Sie ist bspw. blau bei Android und
grau bei der iOS Version einer mit Xamarin.Forms erstellter App.
Obwohl die Ansichten plattformübergreifend im "`shared"' Projekt erstellt werden, werden sie auf
den Zielplattformen nicht gleich aussehen. Grund dafür ist die Tatsache, dass der gemeinsame Code
aus dem "`shared"' Projekt zum nativen Code kompiliert wird. D.h. es entstehen native Steuerelemente, die spezifische Merkmale
der jeweiligen Plattform tragen.
Sollte die Hintergrundfarbe bei Android auch grau sein, wäre ein Custom Renderer im
plattformspezifischen Droid-Projekt nötig. D.h., das native Android-Steuerelement soll überschrieben
werden.
\\Zusammengefasst bedeutet das, dass Anwendungen, bei denen die Geschäftslogik höher
priorisiert wird als optische Effekte der Benutzeroberfläche, sich ganz bequem mit Xamarin.Forms
erstellen lassen können. Unter diese Kategorie fallen die sogenannten
Business(Enterprise)-Anwendungen.
